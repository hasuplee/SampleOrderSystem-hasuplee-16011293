# PRD.md — 반도체 시료 생산주문관리 시스템

원본 요구사항: `system.pdf` (mincoding, 개인과제). 본 문서는 PDF 기능명세를 코드 구현
기준으로 재정리하고, `PoC/` 4종 프로토타입 리뷰에서 발견된 갭·모호점을 보강한 버전입니다.

## 1. 배경

가상 반도체 회사 "S-Semi"는 반도체 시료(Sample)를 생산해 연구소/팹리스/대학 연구실에
납품한다. 주문량 급증으로 엑셀·메모장 관리의 한계(주문 처리 여부 불명확, 재고·공정 현황
파악 어려움)에 부딪혀 콘솔 기반 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 2. 역할(Actor)

| 역할 | 책임 |
|---|---|
| 고객 | 필요한 시료를 이메일 등으로 요청 (시스템 외부 행위, 코드 대상 아님) |
| 주문 담당자 | 요청에 맞게 주문서 작성(시료 주문 메뉴) |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산 처리 |

시스템은 콘솔 기반, 담당자가 직접 메뉴 번호를 입력해 조작한다. 역할별 로그인/권한 분리는
명세에 없음 — 단일 콘솔 프로그램에서 모든 메뉴에 접근 가능하다고 가정한다.

## 3. 주문 상태(OrderStatus)

| 상태 | 의미 |
|---|---|
| `RESERVED` | 주문 접수 (시료 주문 직후 기본값) |
| `REJECTED` | 주문 거절. 정상 흐름 밖의 상태이며 **모니터링/재고계산에서 제외** |
| `PRODUCING` | 승인 완료 + 재고 부족으로 생산 중 |
| `CONFIRMED` | 승인 완료 + 출고 대기 중 (재고 충분했거나 생산이 완료된 경우) |
| `RELEASE` | 출고 완료 |

상태 전이:

```
RESERVED --(거절)--> REJECTED
RESERVED --(승인, 재고충분)--> CONFIRMED
RESERVED --(승인, 재고부족)--> PRODUCING --(생산완료)--> CONFIRMED
CONFIRMED --(출고처리)--> RELEASE
```

## 4. 기능 명세

### 4.1 메인 메뉴

전체 시료 요약(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수)을 표시하고
아래 5개 하위 기능으로 이동한다.

| 메뉴 | 설명 |
|---|---|
| 시료 관리 | 시료 등록 / 목록 조회 / 이름 검색 |
| 시료 주문 | 고객 주문 접수(RESERVED 생성) |
| 주문 승인/거절 | RESERVED 주문에 대한 승인·거절 처리 |
| 모니터링 | 상태별 주문 수, 시료별 재고 현황 확인 |
| 생산라인 조회 | 생산 중/대기 중인 생산 작업 확인 |
| 출고 처리 | CONFIRMED 주문에 대한 출고 실행 |

### 4.2 시료 관리

- **시료 등록**: 입력값 = 시료 ID, 이름, 평균 생산시간(min/ea), 수율(0.0~1.0). 재고는
  등록 시 0 또는 초기값을 입력받는다(PoC 기준 기본 0, 세부는 5절 데이터 모델 참고).
  - 시료 ID는 유일해야 한다 (중복 등록 거부).
- **시료 조회**: 등록된 모든 시료 목록 + 현재 재고 표시.
- **시료 검색**: 이름 부분 일치 검색 (Cycle 1 GREEN 구현 범위).
  - **TODO(백로그)**: ID 부분 일치 검색은 Cycle 1 REVIEW에서 스코프 밖으로 확인되어 보류됨.
    필요 시 별도 사이클에서 `test_ID로_시료를_검색하면_해당_시료가_반환된다` 테스트를 추가하고
    `SampleRepository.search()`가 `sample_id LIKE ? OR name LIKE ?`를 사용하도록 확장한다.
- 수율 = 정상 생산 시료 수 / 총 생산 시료 수 (예: 100개 생산 중 90개 정상 = 0.9).

### 4.3 시료 주문 (예약)

- 입력값: 시료 ID, 고객명, 주문 수량.
- **시료 ID는 반드시 등록된 시료여야 한다** — 존재하지 않는 ID로 주문 시 즉시 거부한다
  (PoC ConsoleMVC에는 이 검증이 없어 승인 단계에서 예외가 발생하는 결함이 있었음 → 반드시 수정).
- 생성 직후 상태는 `RESERVED`.

### 4.4 주문 승인/거절

- RESERVED 상태 주문 목록을 표시하고 특정 주문을 선택해 승인 또는 거절한다.
- **거절**: 즉시 `REJECTED`로 전환.
- **승인**: 대상 시료의 현재 재고와 주문 수량을 비교한다.
  - 재고 충분(재고 ≥ 주문 수량): 주문을 즉시 `CONFIRMED`로 전환.
  - 재고 부족(재고 < 주문 수량): 부족분 = 주문 수량 − 재고. 아래 계산으로 생산 작업(Job)을
    생산 큐(FIFO)에 등록하고 주문을 `PRODUCING`으로 전환.
    - 실생산량 = `ceil(부족분 / 수율)`
    - 총 생산 시간 = 평균 생산시간 × 실생산량
- **재고 차감 시점(확정)**: 주문이 `CONFIRMED`로 전환되는 시점에 재고에서 주문 수량만큼
  차감한다.
  - 재고가 충분해 즉시 CONFIRMED로 전환되는 경우: 승인 처리 그 시점에 차감.
  - 재고 부족으로 PRODUCING을 거치는 경우: 생산완료(4.6절) 시 실생산량만큼 재고를 먼저
    증가시킨 뒤, CONFIRMED 전환과 함께 주문 수량만큼 차감한다.

### 4.5 모니터링

- **주문량 확인**: 상태별(RESERVED/CONFIRMED/PRODUCING/RELEASE) 주문 건수. REJECTED 제외.
- **재고량 확인**: 시료별 현재 재고 + 재고 상태.
  - `고갈`: 재고 0
  - `부족`: 재고 < 대기 수요(미출고 주문 수량 합, RESERVED+PRODUCING 기준 — DataMonitor PoC
    구현 기준. ConsoleMVC PoC는 고정 임계치(<50)를 쓰고 있어 불일치 → **DataMonitor 방식(대기
    수요 대비)으로 통일**한다)
  - `여유`: 재고 ≥ 대기 수요

### 4.6 생산라인 조회

- 생산 라인은 단일 라인, FIFO로 처리한다. 하나의 생산 라인은 한 번에 시료 하나만 생산한다.
- 현재 생산 중인 작업의 정보(주문번호, 시료, 부족분, 실생산량, 진행 상황)를 표시한다.
- 대기 큐(생산 순서 대기 중인 작업 목록)를 표시한다.
- **생산완료 처리(확정)**: 생산라인 조회 메뉴에 "생산완료 처리" 액션을 추가해, 담당자가 현재
  처리 중인 생산 작업(큐의 맨 앞)을 수동으로 완료 처리한다. 완료 처리 시:
  1. 대상 시료 재고를 `actual_qty`만큼 증가시킨다.
  2. 해당 주문을 `CONFIRMED`로 전환하고, 4.4절의 확정된 정책에 따라 주문 수량만큼 재고를
     즉시 차감한다.
  3. 생산 큐에서 해당 작업을 제거하고 다음 대기 작업을 진행 중 상태로 올린다.
  (PoC ConsoleMVC에는 `complete_current_job()` 메서드는 있으나 메뉴/뷰에서 호출되지 않아
  실제로는 도달 불가능한 결함이 있었음 → 반드시 수정.)

### 4.7 출고 처리

- CONFIRMED 상태 주문 목록을 표시하고 특정 주문을 선택해 출고 처리한다.
- 처리 시 상태를 `RELEASE`로 전환한다.

## 5. 데이터 모델

### Sample (시료)

| 필드 | 타입 | 설명 |
|---|---|---|
| sample_id | str (PK) | 시료 ID |
| name | str | 시료명 |
| avg_production_time | float | 평균 생산 시간 (min/ea) |
| yield_rate | float | 수율 (0.0~1.0) |
| stock | int | 현재 재고 수량 |

### Order (주문)

| 필드 | 타입 | 설명 |
|---|---|---|
| order_id | str (PK) | 주문 ID |
| sample_id | str (FK → Sample) | 대상 시료 |
| customer_name | str | 고객명 |
| quantity | int | 주문 수량 |
| status | OrderStatus | RESERVED/REJECTED/PRODUCING/CONFIRMED/RELEASE |

### ProductionJob (생산 작업, 생산 큐 원소)

| 필드 | 타입 | 설명 |
|---|---|---|
| order_id | str | 대상 주문 |
| sample_id | str | 대상 시료 |
| shortage_qty | int | 부족분 = 주문량 − 재고(승인 시점 기준) |
| actual_qty | int | 실생산량 = ceil(부족분 / 수율) |
| total_time_min | float | 총 생산 시간 = 평균 생산시간 × 실생산량 |

영속성: `PoC/DataPersistence`가 검증한 SQLite 스키마(samples/orders 테이블)를 그대로
계승한다. ID 채번은 메모리 카운터가 아니라 DB 기준 "기존 최댓값 + 1" 방식(DummyDataGenerator
PoC 방식)을 따른다.

## 6. 확정된 설계 결정 / 남은 미해결 사항

아래 4가지는 사용자 확인을 거쳐 확정되었습니다(결정 이력은 `Plan.md` 참고, 본문은 4절에 반영됨).

1. **재고 차감 시점**: 주문 승인(CONFIRMED 전환) 시점에 차감 — 확정.
2. **생산완료 트리거 방식**: 담당자 수동 처리(생산라인 메뉴의 "생산완료 처리" 액션) — 확정.
3. **기존 PoC 코드 처리 방침**: `PoC/`는 보존, 최종 구현은 `src/` 아래 신규 작성 — 확정.
4. **개발 의존성 정책**: 런타임은 표준 라이브러리만, pytest/ruff 등 개발 도구는 허용 — 확정.

남은 미해결 사항:

5. **Chapter 3(미션 소개 및 제출 방법)**: `system.pdf`에 목차만 있고 본문이 없음(23페이지

## 7. PoC 리뷰에서 발견된 갭 (최종 구현 시 반드시 해소)

- 존재하지 않는 시료 ID로 주문 생성 시 검증 없음 (승인 단계에서 예외 발생) → 4.3에서 해결.
- 재고 증감 로직 전체 부재 → 6-1 확정 후 구현.
- 생산완료 처리 메뉴/트리거 부재(`complete_current_job()` 도달 불가) → 6-2 확정 후 구현.
- 입력값 검증 부족(숫자 외 입력 시 프로그램 예외 종료) → 모든 사용자 입력에 검증/재입력
  루프 적용.
- 재고 상태 판단 기준이 PoC마다 다름(ConsoleMVC: 고정 임계치 vs DataMonitor: 대기수요 대비)
  → 4.5에서 DataMonitor 방식으로 통일.
- 메모리 기반 카운터 ID 채번(`len(orders)+1`) → DB 기준 채번 방식으로 교체.
